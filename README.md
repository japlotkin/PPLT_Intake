# PPLT Intake Dashboard

Internal dashboard for Pinder Plotkin Legal Team and Abogado Attorney.
Aggregates GoHighLevel (PPLT Leads + Abogado Attorney locations) and
Meta Ads (PPLT, Workers' Comp, Abogado ad accounts) into a single
read-only view of leads, referrals, signed cases, and intake activity.

## Stack

- Next.js 16 (App Router) + Tailwind 4
- Recharts (bar + pie)
- Clerk for per-user auth (admin role = `jplotkin@pinderplotkin.com`)
- bcrypt-protected firm-wide gate in front of Clerk
- Vercel KV hourly cache; manual "Refresh" button to bust

## Layout

```
src/
  middleware.ts            site-wide gate + Clerk auth
  app/
    layout.tsx             ClerkProvider wrapper
    page.tsx               -> redirects to /dashboard
    gate/page.tsx          firm password page
    sign-in/[[...rest]]    Clerk sign-in
    dashboard/page.tsx     the dashboard (single scrollable page)
    api/
      gate/route.ts        POST password, set HMAC-signed cookie
      data/route.ts        GET dashboard payload (cached 1h)
      refresh/route.ts     POST cache bust (admin only)
  components/              StatCard, KpiTable, DeltaPill, charts
  lib/
    env.ts                 typed env reading
    gate.ts                bcrypt + HMAC cookie (Web Crypto, edge-safe)
    cache.ts               Vercel KV with in-process fallback
    dateRanges.ts          Mon-Sun weeks, calendar months/quarters (America/New_York)
    mapping.ts             loads mapping.json + classifier helpers
    ghl/                   GHL v1+v2 client, opportunities, contacts, conversations, reviews, emails
    meta/                  Meta Graph API insights client
    metrics/               overview, kpi, leadAnalytics, intakeTeam, caseAnalytics, email
mapping.json               discovered pipeline/stage/tag/field structure for both GHL locations
```

## mapping.json

`mapping.json` is the source of truth for pipeline/stage classification.
It was generated by `_discover_ghl_mapping.py` (in the parent Claude Context
directory) and committed for review. To regenerate:

1. Run the discovery script with up-to-date GHL credentials.
2. Diff `mapping.json`; edit any pipelines/tags/fields the auto-classifier
   got wrong (purpose, practice_area, co_counsel_firm).
3. Commit and redeploy — the dashboard reads it at request time.

The associated `mapping-review.md` summarizes what was auto-detected.

## Local dev

```bash
cp .env.example .env.local
# fill in GHL, Meta, Clerk, and SITE_GATE_* values
npm install
npm run dev
# open http://localhost:3000 (or 3030 if you use the launch.json)
```

The firm password is gated by `SITE_GATE_PASSWORD_HASH` (bcrypt). To set
or rotate it:

```bash
node -e "console.log(require('bcryptjs').hashSync('YOUR_PASSWORD',12))"
```

Paste the resulting hash into `SITE_GATE_PASSWORD_HASH` and rotate
`SITE_GATE_COOKIE_SECRET` to invalidate existing gate sessions.

## Production deploy

Vercel connects to this repo. Each push to `main` deploys.

Required environment variables on Vercel (Project → Settings → Environment
Variables): everything in `.env.example` plus the KV pair provisioned by
Vercel KV when you link a KV store to the project.

The dashboard runs in Node.js runtime (not Edge) so it can fetch from GHL
+ Meta with full timeouts.
